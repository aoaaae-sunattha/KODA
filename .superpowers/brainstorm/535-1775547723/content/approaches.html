How should we present KODA to non-tech interviewers?

Pick the approach that best fits your interview style — or mix ideas from multiple options

A

Story-First README

Narrative arc: "I built an app → wrote a test plan → automated it → wired up CI/CD."
README tells the story in plain English with a visual testing-cycle diagram. One HTML "showcase" page embedded as a link — shows screenshots + numbered steps. Best for interviewers who read top-to-bottom.

  • ✅ Easy to scan, no tech jargon
  • ✅ Clear "why I did this" story
  • ⚠️ Requires interviewer to click around the repo
B

Self-Contained HTML Showcase Page

One file, everything inside: An HTML page that is the presentation. Sections: What is KODA → The Testing Cycle (with animated steps) → Coverage Dashboard (numbers) → CI/CD flow diagram → How to run it. README becomes a short index pointing here.

  • ✅ Interviewer opens one file — done
  • ✅ Visual, polished, impressive
  • ✅ Screenshots embedded directly
  • ⚠️ Slightly heavier to build
C

Dual-Layer: README + Showcase Page

Best of both: README is completely rewritten for non-tech readers with a clear testing-cycle timeline, quick-start commands, and links to key files. A separate showcase/index.html is a visual portfolio page with screenshots, coverage chart, and CI/CD diagram — linked prominently at the top of the README.

  • ✅ README works standalone for quick readers
  • ✅ HTML page impresses visual thinkers
  • ✅ Both non-tech and tech interviewers are served
  • ⚠️ Two things to maintain (minor)

My recommendation

Option C — since the interviewer is non-tech but "might be hands-on to the tech side," you want both layers. The README gives the story immediately; the showcase page gives visual proof. anyday.io is a tech company — a well-made HTML page signals frontend awareness even in a QA role.
